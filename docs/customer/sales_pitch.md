## 1. エレベーターピッチ（一言）

**[製品名] は、データ分析チーム 1 部門分の機能を「ソフトウェアとして」導入できる、エンタープライズ向け AI 分析プラットフォームです。**

PM、SQL エンジニア、統計家、QA レビュアー、ビジネスコンサルタントの役割を持つ AI エージェントが連携し、顧客社内データに対して、課題定義から最終提案までを一貫して遂行します。生成 AI 単体ではなく、品質保証・監査性・再現性を中核に据えた「業務として使える」分析自動化基盤です。

**60 秒バージョン:**

> 多くの企業は「データはあるのに分析者が足りない」「コンサルに頼むと遅くて高くて、結果を社内で再現できない」「ChatGPT に分析させるのはハルシネーションが怖い」という三重苦を抱えています。[製品名] はこの 3 つを同時に解きます。役割分担された複数の AI エージェントが、計画レビューと多層 QA を経て分析を進め、すべての数値主張に出典を紐付け、データスナップショットと共に成果物を保存します。コンサル費用の数分の一で、いつでも、再現可能な形で、御社の意思決定を支援します。

## 2. ターゲット顧客

### 2.1 業種

- **EC・小売**: 顧客分析、チャーン、施策効果、在庫最適化
- **SaaS・サブスクリプション**: 製品エンゲージメント、価格戦略、解約予測
- **金融**: 顧客 LTV、不正検知、ポートフォリオ分析
- **製造**: 需要予測、サプライチェーン分析、品質データ分析
- **メディア・通信**: コンテンツパフォーマンス、解約防止
- **不動産・人材**: マッチング分析、市場動向

### 2.2 顧客の役職

- **データ・アナリティクス責任者** (CDO / VP of Data / Head of Analytics)
- **経営企画・事業企画責任者** (CSO / VP of Strategy)
- **DX 推進責任者**
- **CTO / VP of Engineering**（社内データ基盤を持っている前提）

### 2.3 典型的な課題（ペインポイント）

| ペイン | 顧客の声 |
|---|---|
| 分析依頼の積み上がり | 「依頼から結果まで 2-4 週間。営業や経企が待ちきれず勘で動く」 |
| 分析者の採用難 | 「シニア分析者は年収 1,500 万でも採れない。育てるのに 2-3 年」 |
| コンサル費用の高さ | 「1 案件 500 万円〜数千万円。しかも結果を社内で再現できない」 |
| 過去分析の属人化 | 「前任者が辞めたら指標定義もロジックも分からなくなった」 |
| AI への不信 | 「ChatGPT に出させた数字を経営会議に出すのは怖い」 |
| 説明責任 | 「監査・株主から『この数字の出し方は』と聞かれて答えられない」 |
| BI ツールの限界 | 「グラフは出るけど『なぜ』『次に何をすべきか』が出ない」 |

> **営業ヒント**: 初回商談では、技術説明より先にこの表のどれが顧客に当てはまるかを確認する。「御社の場合、どれが一番大きい痛みですか?」と問うと、商談が深まる。

## 3. 製品概要

### 3.1 何を提供するか

[製品名] は、以下のような **「役割分担された AI エージェントチーム」** を貴社のデータ環境に対して提供します。

| エージェント | 人間に例えると |
|---|---|
| Supervisor | プロジェクトマネージャー / リードコンサルタント |
| Scoping Specialist | 課題定義担当 / 分析企画担当 |
| Metric Definition Specialist | 指標設計担当 / KPI コンサルタント |
| SQL Specialist | データエンジニア / SQL エキスパート |
| Python Specialist | データサイエンティスト |
| Causal / Experimentation Specialist | 因果推論・実験計画の専門家 |
| QA Specialist | 分析品質レビュアー |
| Insight Specialist | 戦略アナリスト |
| Business Recommendation Specialist | ビジネスコンサルタント |
| Advisor | シニアコンサルタント（壁打ち相手） |
| State Manager | プロジェクト事務局 / ナレッジ管理 |

これらが **チームとして連携し、人間の確認ポイントを挟みながら** 分析プロジェクトを遂行します。

### 3.2 どう動くか（顧客向け 7 ステップ）

1. **要件すり合わせ** — 質問・対話で目的と成功条件を固める
2. **論点整理と指標定義** — 何を測るか、どう定義するかを明確化
3. **データ理解** — 社内テーブルを把握、品質懸念を確認
4. **分析実行** — SQL / Python による集計・統計・可視化
5. **品質保証** — 3 段階の独立 QA レビュー
6. **示唆抽出** — 結果から「なぜ」「だから何」を導出
7. **提案化** — 経営層・実務担当向けの提案ドキュメント生成

各フェーズで中間生成物が UI に表示され、貴社が **承認・差し戻し・追加要望** を出せます。

## 4. コア価値提案（5 つの強み）

### 4.1 「それっぽいだけ」を防ぐ多層品質保証

**顧客の不安**: AI に分析させると、それっぽい数字を捏造されるのが怖い。

**本製品の答え**:

- **計画レビュー (Plan-Execute)**: 重い SQL や Python を実行する前に、計画段階で参照するテーブル・カラム・手法を Supervisor が承認。無駄な実行・誤った前提を排除。
- **3 段階 QA**: ①データ品質、②集計妥当性、③主張の根拠の独立検証。各段階を通過しないと次に進めない。
- **数値主張の出典必須化**: 最終レポートのすべての数値に、根拠となる集計結果（行・列・集計キー）が紐付く。
- **存在しないカラム参照を物理的に拒否**: SQL 実行前にデータベーススキーマと照合、未知のカラム参照は実行段階で拒否。

> **営業ヒント**: 「ChatGPT で試したけど嘘の数字が出てきて懲りた」という顧客には、ここを最初に出す。

### 4.2 データ分析チーム 1 部門分の機能を即日立ち上げ

**顧客の不安**: シニアアナリスト採用に 6 ヶ月、教育に 1 年、それでも辞めるかも。

**本製品の答え**:

- 11 種類の専門エージェントが揃った状態で、**契約直後から稼働開始**
- 1 人の汎用 LLM ではなく、**役割ごとに最適化された専門家チーム**
- 並列処理で、人間チームの数倍速で複数案件を処理
- 24 時間 365 日稼働、深夜・週末の急ぎ案件にも対応

> **営業ヒント**: 採用コスト・育成コスト・退職リスクを定量化して比較表を作ると刺さる。

### 4.3 すべての分析が監査可能・再現可能

**顧客の不安**: 数ヶ月後、規制当局や経営会議で「この数字どう出した？」と聞かれた時に答えられない。

**本製品の答え**:

- **データスナップショット固定**: 分析時点のデータ状態を pin、後から同じ条件を再現可能
- **完全な意思決定ログ**: どの AI が、いつ、何を入力に、何を出力したかを完全記録
- **アーティファクトの系譜管理**: 最終結論から元データまで、すべての中間生成物を辿れる
- **プロンプト・モデル設定のバージョン管理**: 同じ設定で再実行すれば同じ結果が出る設計

> **営業ヒント**: 上場企業・規制業界（金融・医療）には特に強く刺さる。「内部監査・SOX・株主説明に耐える分析基盤」という言い方を。

### 4.4 顧客の業務知識が資産として蓄積される

**顧客の不安**: 過去の分析が属人化、退職や担当変更で全部ゼロから。

**本製品の答え**:

- **指標レジストリ**: 「アクティブユーザー」など、社内で確定した指標定義をライブラリ化、次回案件で自動参照
- **過去案件知見の検索**: 過去の分析・落とし穴を、新案件で必要時に明示的に参照
- **意思決定ログの永続化**: 「なぜこの判断をしたか」が消えない
- **データカタログとの統合**: 社内テーブル仕様書を AI が参照、属人的な「あの人だけが知っている」を解消

### 4.5 人間の判断を必要な場面に集中させる設計

**顧客の不安**: AI に丸投げすると暴走するし、全部チェックしたら結局人間がやるのと同じ。

**本製品の答え**:

- **Justified Override 設計**: AI も人間も、ルールを上書きするには根拠と緩和策の明示が必要
- **リスクベースの承認分岐**: 低リスク作業は自動承認、中・高リスクのみ人間判断
- **ヒューマンインザループのフェーズゲート**: 重要節目で必ず人間に確認
- **Plan-Reviewer / Phase-Gate 自動評価**: 形式チェックは AI が前処理、人間は本質的判断に集中

## 6. ユースケース集

### 6.1 EC: チャーン分析と打ち手提案

**シナリオ**: 「直近 3 ヶ月のリピート率が落ちている。原因と打ち手を出してほしい」

**本製品の動き**:

1. Scoping が「リピート率」の定義を確認、3 候補を提示してユーザー承認を取る
2. SQL Specialist がコホート別リピート率を集計、QA が定義整合をチェック
3. Causal Specialist が「新規流入チャネル変化が交絡因子になっていないか」を検証
4. Insight Specialist が「Q2 開始の送料改定が SMB 顧客のリピートに影響」と仮説提示
5. Business Recommendation Specialist が打ち手を 3 案提示、優先順位付け

**従来比**: 通常 2-3 週間 → **半日〜1 日**

### 6.2 SaaS: 製品エンゲージメント診断

**シナリオ**: 「主要機能ごとのエンゲージメントを可視化、解約予測モデルを作ってほしい」

**本製品の動き**:

1. Metric Definition がエンゲージメント指標を 5 軸で定義
2. SQL Specialist が機能別利用ログを集計、Python Specialist が時系列で可視化
3. Python Specialist が解約予測モデルを構築、QA が再現性を検証
4. Causal Specialist が「機能 X の利用減少が解約の先行指標か」を検定
5. Insight が「アクセス頻度より深さがリスク指標」と提示
6. Reporting が経営層向け 1 ページサマリと詳細レポートを生成

### 6.3 金融: クレジットリスク要因分析

**シナリオ**: 「住宅ローン審査の延滞率が上昇、要因を分析」

**本製品の動き**:

1. Scoping が「延滞」の定義（30 日 / 60 日 / 90 日）を確認
2. SQL が借入属性 × 延滞ステータスをセグメント集計
3. Python Specialist が決定木と SHAP で要因寄与を可視化
4. QA Specialist が **3 段階 QA を完全実施**（規制業界のため）
5. すべての結論にデータスナップショット ID と監査ログが紐付き、内部監査・金融庁検査に提示可能

**強み**: 監査対応コストが激減、規制業界の説明責任を仕組みで担保。

### 6.4 製造: 需要予測モデルの精度劣化調査

**シナリオ**: 「6 ヶ月前に作った予測モデルの精度が落ちている。何が変わったか調査」

**本製品の動き**:

1. State Manager が過去案件の指標定義・前提を呼び出し
2. SQL Specialist が当時のデータと現在のデータを比較
3. Python Specialist が特徴量分布のドリフトを検出
4. Causal Specialist が「コロナ後の需要パターン構造変化」を確認
5. Insight が「3 つの新セグメントを切り分けて再学習推奨」

**強み**: 過去案件の前提が消えていないので、調査が高速。

## 8. セキュリティ・ガバナンス・コンプライアンス

> エンタープライズ商談では避けて通れないセクション。情シス・法務・監査部門への説明素材として使う。

### 8.1 データ取り扱い

- **テナント完全分離**: 顧客データは他顧客と論理的・物理的に分離
- **オンプレ・プライベートクラウド対応**: 顧客 VPC 内・オンプレに deploy 可能（プラン依存）
- **データは顧客環境を出ない構成オプション**: SQL 実行は顧客 DWH 上、結果メタデータのみ本製品が管理
- **PII / 機密データ列の自動マスキング**: アクセス前に redaction layer を通過

### 8.2 アクセス制御

- 最小権限の原則（agent ごと、案件ごと、テーブル・カラムレベルで制御）
- すべてのツール呼び出しが監査ログに記録される
- ロールベースの View アクセス制御（特定の AI agent は特定の State View のみ参照可能）

### 8.3 監査・コンプライアンス

- **完全な意思決定ログ**: 誰が（どの AI が）、いつ、何を入力に、何を出力したか
- **データスナップショット ID**: 分析時点のデータ状態を再現可能
- **プロンプト / モデルバージョン管理**: モデル変更があっても過去結果を再現可能
- **OpenTelemetry 互換のトレーシング**: 既存の SIEM / 監視基盤に統合可能

### 8.4 想定対応規制

- SOC2 / ISO 27001（取得計画 or 取得済みを記載）
- GDPR / 個人情報保護法
- 金融業界向け FISC 安全対策基準（オプション）
- 医療業界 3 省 2 ガイドライン（オプション）

> **営業ヒント**: 規制業界の場合、「監査可能性」「再現性」「データ越境ゼロ構成」の 3 点を最初に必ず提示。

## 9. 営業 FAQ・反論対応

### 9.1 製品仕様への反論

**Q: AI が間違った分析を出したらどう責任を取るのか?**

A: 単一の AI が出す結論ではなく、複数の専門エージェントが計画レビュー・3 段階 QA を経て出す結論です。すべての数値主張に出典が紐付き、推論プロセスが完全にログ化されているため、問題が起きても「どこで何が起きたか」を即時特定できます。さらに、最終提案前に必ずユーザー承認のステップを挟むため、人間の最終判断が常に介在します。

**Q: 既存の BI ツール（Tableau / Looker）と何が違うのか?**

A: BI ツールは「データを見せる」道具で、誰かが集計ロジックを定義する必要があります。本製品は「定義から集計、解釈、提案までを行う AI 分析チーム」です。BI ツールのダッシュボードに対して「なぜこの数字なのか?」という問いに答えるのが本製品の役割で、競合ではなく補完関係です。

**Q: ChatGPT や Claude を直接使えば安いのでは?**

A: チャット型 AI は会話には優れますが、(1) 社内データに安全にアクセスする仕組み、(2) ハルシネーション抑止、(3) 監査ログ、(4) 役割分担された複数エージェントの協調、(5) 過去知見の蓄積、これらすべてを自社で構築する必要があります。本製品は「業務として使える」状態に組み上げた完成品です。

**Q: シニアデータサイエンティストの判断と同等のことができるのか?**

A: トップ 10% のシニアの「ひらめき」までは代替できません。しかしジュニア〜ミドルレベルの定型・準定型分析の 8 割は本製品が代行可能です。シニアは戦略判断に集中、本製品が量を捌く、という分業をご提案しています。

### 9.2 セキュリティへの反論

**Q: うちのデータを外部の AI に投げるのは情シスが許さない**

A: 構成オプションが複数あります。最高セキュリティのプランでは、顧客 VPC 内に deploy し、SQL 実行は顧客 DWH 内で完結、本製品側にはメタデータと意思決定ログのみが残ります。データ本体が外部に出ない構成が選べます。

**Q: AI モデルが顧客データで学習されるのでは?**

A: 顧客データはプロンプトとしてモデルに渡しますが、**モデルの再学習には使用しません**。モデルはステートレスに実行され、案件間で情報が漏れることはありません。複数顧客の知見統合は明示的な opt-in でのみ行います（ベンチマーク機能等）。

**Q: 監査の時にどう説明すれば?**

A: 全ての分析プロセスがログ化されており、「いつ、どの AI が、どのデータスナップショットを、どのプロンプト version で参照し、どんな結論を出したか」を完全に再現できます。Excel と PowerPoint の組み合わせよりも、はるかに監査適格性が高い基盤です。

### 9.3 価格・契約への反論

**Q: 高すぎる**

A: 比較対象を確認させてください。外部コンサル 1 案件 500 万円・年 5 案件で 2,500 万円、データサイエンティスト 1 名年俸 1,500 万円・採用コスト 300 万円。これらに対して本製品は……（具体的な価格を提示）。さらに採用リスク・離職リスク・案件待機リスクが消えます。

**Q: PoC からスタートしたい**

A: 推奨です。1-2 ヶ月の PoC で、御社の既存案件を 3-5 件、本製品で再現します。リードタイム短縮率・品質スコア・総コストを実測した上で本契約をご判断ください。

**Q: ベンダーロックインが心配**

A: アーティファクト（分析結果）は標準フォーマット（JSON / Parquet）でエクスポート可能。指標定義もテキストで持ち出し可能。プロンプトはお客様のものとして契約書に明記します。

### 9.4 組織導入への反論

**Q: うちのデータチームが反発するのでは?**

A: よくある懸念です。本製品は「分析者を置き換える」のではなく「分析者の生産性を 5-10 倍にする道具」と位置付けています。退屈な集計作業から解放され、戦略判断と高度な分析に集中できる、というメッセージで導入された企業が多いです。導入時のチェンジマネジメントもご支援します。

**Q: 現場の人が使いこなせるか不安**

A: チャット形式の自然言語インターフェースなので、SQL も Python も書けない方でも使えます。各フェーズで AI が中間生成物を提示し、ユーザーが「これで OK」「ここを変えて」と返すだけで進みます。

## 13. 補足: 営業として絶対にやってはいけないこと

> 製品の信頼性を守るためのガードレール。

- **ハルシネーション率 0% と言わない** — 「多層対策で実用上問題ないレベルに抑える」が正しい表現
- **AI が完全自動で分析する、と言わない** — 「人間の確認ポイントを必ず挟む設計」が正しい
- **シニア分析者を完全に置き換える、と言わない** — 「シニアは戦略判断に集中、本製品が量を捌く」分業モデル
- **競合製品を悪く言わない** — 「補完関係」「役割が違う」という言い方を徹底
- **未確認の事例・数字を出さない** — すべての引用には根拠が必要
- **顧客データの取り扱いについて曖昧な約束をしない** — 法務確認が必要な事項は持ち帰る
